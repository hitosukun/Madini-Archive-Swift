# セキュリティポリシー

[English version](./SECURITY.en.md)

Madini Archive は LLM の会話ログをローカル `archive.db` に統合して読み返すためのアプリです。会話ログには直接的な機密情報だけでなく、**間接的に個人特定や推測材料になりうる情報** が大量に含まれます。本ドキュメントはコントリビュータ・改造者・企業内利用者が、自分自身と会話に登場する第三者のプライバシーを守るために最低限知っておくべき方針を記録します。

## 脆弱性の報告

本リポジトリのコード自体に脆弱性を発見した場合は、GitHub の **Security Advisories** 機能（リポジトリの "Security" タブ → "Report a vulnerability"）から非公開で報告してください。public な issue として投稿しないでください。

## What counts as "personal data" in conversation logs

LLM の会話ログを扱うとき、以下はすべて「個人データ」として扱います。リポジトリ・issue・コミット・PR・テストデータ・ログ出力・スクリーンショットなど、**公開される可能性のある任意の場所** に混入させないでください。

### 直接的な機密情報（明らかに守るべきもの）

- パスワード、API キー、アクセストークン、認証情報全般
- クレジットカード番号、銀行口座番号、暗号資産のシードフレーズ・秘密鍵
- マイナンバー、社会保障番号、運転免許証番号、パスポート番号
- 医療情報、診断結果、処方薬、検査結果

### 間接的な推測材料（軽視されがちだが攻撃材料になる）

これらは単独では「個人情報」と認識されにくいですが、組み合わせるとパスワード推測・秘密の質問の突破・フィッシング・SNS 工学攻撃・doxxing（個人特定）の起点になります。

- 生年月日、誕生日、年齢
- 本名、旧姓、ニックネーム
- 家族の名前、ペットの名前
- 居住地、出身地、勤務地、学校名
- 卒業年、入社年
- 電話番号、メールアドレス
- SNS のハンドルネーム（複数アカウント間の紐付け材料になる）
- 顔写真、自撮り画像
- 趣味、好きな作品、繰り返し言及する固有名詞（パスワード推測材料になる）
- 訪問頻度の高い場所（自宅・職場の特定材料になる）
- 仕事の取引先、社内システム名、プロジェクト名
- 体調、生活リズム、不在時間帯（物理的なセキュリティリスクに繋がる）

### 会話の文脈そのもの

会話の話題そのものが、本人または会話相手のセンシティブな個人情報を構成することがあります。

- 思想・宗教・政治的見解
- 性的指向、ジェンダー、性自認
- 病歴・通院歴・メンタルヘルスの相談内容
- 借金、税金、金銭トラブルの相談
- 法的トラブル、訴訟、刑事事件への関与
- 家族関係の問題、恋愛、不倫
- 仕事の愚痴、上司や同僚の固有名詞を含む文句
- 依存・嗜癖の告白

これらが含まれた会話ログは、断片的にでもリポジトリに混入すると、所有者本人だけでなく **会話に登場する第三者** のプライバシー侵害につながります。会話相手は当事者として同意していない可能性が高いことに注意してください。

## コーディングAI への注意

Claude Code / Cursor / Codex 等のコーディングAI は効率を優先するため、以下のような提案をしてくることがあります。**これらはすべて却下してください**：

- 「実際の `archive.db` からテストデータを抜粋して fixture にしましょう」
- 「エラー再現のため、実際のメッセージを issue に貼ってください」
- 「ユーザー設定の確認のため、preferences の中身を共有してください」
- 「デバッグログに会話本文を出力するようにしましょう」
- 「再現手順として、起こった会話の流れをそのまま記述してください」
- 「fixture として、最近の会話ログから 5 件抽出しましょう」

これらは一見合理的ですが、プライバシーリスクを評価していません。次のように明示的に指示してください：

- **合成データを使う**（"Generate fully synthetic conversations that demonstrate the schema, with placeholder names like Alice / Bob"）
- **ID で参照する**（"Reference conversation by `conv_id`, do not paste the message body"）
- **伏せ字にする**（"Replace any actual names, places, dates with `[REDACTED]` or generic placeholders before pasting"）
- **アーカイブ全体ではなく構造のみ**（"Show me the SQL schema and a representative empty row, not actual data"）

## 企業利用について

このアプリのコードベースを企業内で改造する場合、社内ガイドラインで以下を明確化することを強く推奨します：

- **業務関連の LLM 会話ログは個人情報・機密情報として扱う。** 業務上の判断、取引先との打ち合わせ、未公開プロジェクトの議論、人事・労務関連の相談などが含まれる
- **改造作業は業務 PC で行い、private リポジトリで管理する。** 個人用 GitHub アカウントへの fork や、業務外の環境への持ち出しを禁止する
- **public リポジトリへの push は二人以上の目視確認を必須とする。** PR レビューと別に、機密情報・間接識別子の混入チェックを目視で行う
- **新入社員・インターン・外部委託者は、セキュリティ研修後に作業権限を付与する。** 研修には本ドキュメントの内容を含める
- **退職時にローカル `archive.db` を消去する。** 業務 PC の return 時に、Application Support / 各 LLM プロバイダのキャッシュ・ダウンロード履歴を含めて削除確認する

## 自分のリポジトリを公開する前のチェックリスト

個人プロジェクトとして自分の archive をベースに改造したものを public にする場合：

- [ ] `archive.db` および `*.sqlite*` が `.gitignore` に含まれている
- [ ] `~/Library/Application Support/Madini Archive/` 配下のファイルが `.gitignore` でカバーされている
- [ ] git history を `git log --all` で全コミット message を確認、個人情報を含む文言がないか
- [ ] `git log --all --diff-filter=D` で削除済みファイルを確認、過去に commit した機密ファイルが履歴に残っていないか
- [ ] `git grep` でリポジトリ全体に絶対パス・実名・電話番号・メールアドレスが残っていないか
- [ ] テストフィクスチャー（`Sources/Fixtures/` など）の内容が完全に合成データであるか
- [ ] スクリーンショットを README や docs に貼っている場合、実会話のタイトル・本文・ユーザー名が映り込んでいないか

git 履歴に一度入った機密情報は、`git filter-repo` 等で書き換えるか、リポジトリを作り直さない限り恒久的に残ります。**push 前のチェックが、push 後の対処より圧倒的に安価** です。
